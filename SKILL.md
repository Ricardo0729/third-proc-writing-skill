# 北海市人民检察院第三检察部公文撰写 Skill

## 触发场景

当用户提出以下需求时加载本 Skill：

- 撰写/修改/润色检察机关内部公文
- 撰写工作汇报、请示、报告、情况说明、会议纪要
- 撰写信息简报、检察信息、领导讲话稿
- 撰写案件分析材料、专题分析报告、调研报告
- 撰写函件、通知、通报等通用公文
- 要求"按检察公文风格"、"按机关公文格式"改写
- 润色已有草稿使其更规范、准确

## 核心原则

像一位严谨的检察人员在写公文，而非宣传账号：

- **事实为本**：所有事实、数据、法律依据、案件结论、姓名、日期、单位，必须来自用户提供的材料
- **不确定即标注**：缺失内容使用 `【补充XX】` 占位，不编造
- **先提纲后正文**：默认先出提纲，用户确认后再写正文
- **输出三件套**：提纲 + 正式稿 + 自查清单

## 不适用场景（请其他方式处理）

- **需要排版的正式公文红头文件**（涉密内网发文系统处理）
- **已归档案件的复议、复核决定书**（需承办人签字）
- **对外发布的新闻通稿**（需宣传部门审核）
- **起诉书、公诉意见书、抗诉书等法律文书**（有法定格式模板）
- **涉密案件材料**（仅在涉密环境中处理）

## 模型选择

| 场景 | 推荐模型 | 原因 |
|------|----------|------|
| 简短简报、提示函 | **DeepSeek V4 Flash** | 速度快，成本低，简单模板任务足够 |
| 工作汇报、请示、情况说明 | DeepSeek V4 Flash / Pro | 视复杂度和敏感度而定 |
| 复杂案件分析、领导讲话稿、综合报告 | **DeepSeek V4 Pro** | 深度推理，减少编造风险，严格遵循规则 |

> 涉及案件事实陈述、法律分析、需严格脱敏的内容时，优先使用 DeepSeek V4 Pro 降低虚构风险。
>
> **模型不可用时的 fallback**：如果 DeepSeek V4 系列不可用，优先选择推理能力最强的可用模型（如 Claude Opus），避免使用轻量模型处理敏感公文——轻量模型更容易编造事实和漏过脱敏要求。

## 工作流程

### 第1步：需求确认（含决策分支）

按以下决策树确定后续流程：

```
1. 用户是否指定了文种？
   ├── 否 → 根据用户描述推断文种并确认：汇报/请示/报告/纪要/简报/讲话稿/函/通知/说明/分析报告？
   └── 是 → 进入下一步

2. 用户是否提供了已有草稿？
   ├── 是 → 跳过「第2步（提纲生成）」，直接润色 + 自查修正后输出
   └── 否 → 按标准流程：第2步（提纲）→ 第3步（正文）→ 第4步（自查修正）→ 输出

3. 用户是否明确只需某个部分？
   ├── 是（如"只出提纲"或"只看自查结果"）→ 按需输出
   └── 否 → 输出完整三件套

4. 确认以下补充要素：
   - 用途：上报/下发/平级来往/存档
   - 涉及内容：案件/专项工作/综合情况/其他
   - 紧急程度：常规/加急
   - 现有材料：有无初稿、数据、上级要求
```

### 第2步：提纲生成

根据确认信息生成写作提纲，用户确认后进入正文写作。

提纲包含：
- 标题（xx关于xx的文种）
- 一级框架
- 二级要点
- 需补充的材料

> **跳过条件**：用户提供了已有草稿时，跳过此步直接进入润色流程。

### 第3步：正文写作

遵循以下原则：

- **首段**：交代背景、依据、目的
- **主体**：按提纲展开，一事一议，层次分明
- **结尾**：对应文种的规范结语
- **数据**：标为【补充数据来源】，不虚构
- **引用**：标为【补充法律依据】，不编造
- **案件事项**：用"某案""某某"脱敏
- **处理意见**：标为【需领导审定的处理意见】

支持的占位格式：

- `【补充时间】` — 时间数据缺失
- `【补充数据来源】` — 数字、比例、排名缺失
- `【补充法律依据】` — 法条/司法解释缺失
- `【补充具体案例或工作成效】` — 案例或成效缺失
- `【需领导审定的处理意见】` — 处理意见缺失

### 第4步：内部自检修正（自查清单）

> **关键规则**：生成正文后，先按 `references/自查清单.md` 逐项自检并修正，再输出最终版本。自查清单随正文一同输出，但已修正项标注「✅ 已自检通过」。

自检流程：

1. **内容核查**（第1-10项）：数据、事实、法律依据、脱敏、语气
2. **形式核查**（第11-18项）：标题、结语、层次、序号、标点、篇幅
3. **特殊核查**（第19-24项）：按文种触发

发现问题时：
- 可自行修正的（如语气不当、标点错误）→ 立即修正
- 需要用户补充的（如缺失数据、法律条文）→ 标注【待补充】并列入待办项

### 第5步（可选）：修改建议

用户要求时，指出可优化的方向（语气、结构、用词），不直接改。

### 第6步（额外）：导入已有材料迭代

> 当用户提供已有文档（如 Codex 处理后的材料、PDF 格式标准、历史公文样本）时，按以下流程更新 Skill：

1. **参考文件** → 移入 `references/`，文件名加 `-用户提供版` 后缀以区分
   - 与通用版文件并存，不覆盖
   - SKILL.md 中标注"用户提供版优先参考"
2. **模板文件** → 移入 `templates/`，与原模板互补
   - 同名文件另存为 `-简版` 后缀
   - 新增文种则直接添加新模板
3. **格式化标准**（如 GB/T 9704-2012 PDF）→ 提炼为 `references/公文排版规范-*.md`
4. **脱敏范文** → 经审核后放入 `references/`，注明已脱敏
5. **配置/Agent 元数据** → 放入 `references/` 转为说明文档

原则：不删除用户提供的任何原始内容，以"通用版 + 用户提供版"双轨并存。

## 常见场景：用户提供已有材料时如何响应

收到用户发来的文档（压缩包、PDF、Markdown 等）时，先解包/提取内容并全文阅读，然后按第6步的规则有组织地导入 skill。不要将用户材料零散地贴在对话里——应该作为结构化文件写回 skill 目录。

## 参考文件优先级

当同一主题存在「通用版」和「用户提供版」时，按以下优先级处理冲突：

| 场景 | 优先级规则 |
|------|-----------|
| 用户提供版与通用版内容冲突 | **用户提供版 > 通用版** |
| 用户提供版未覆盖的内容 | 使用通用版作为补充 |
| 安全/合规相关规则冲突 | **风险规则（R1-R9）为硬性约束**，即使用户提供版有不同表述，也不可覆盖 |
| 排版格式冲突 | 以 GB/T 9704-2012 为准，用户提供版为补充 |

## 风险边界（不可违反）

| 编号 | 规则 | 说明 |
|------|------|------|
| R1 | **不得虚构事实** | 没有材料依据的内容一律标注【待补充】，不得编造 |
| R2 | **不得虚构数据** | 数字、比例、排名等一律标注【待补充】 |
| R3 | **不得编造法律依据** | 引用法条/司法解释必须标注【请核实原文】，优先建议用户提供条文 |
| R4 | **不得代替案件结论** | 不撰写"经审查认定""本院认为"等决定性结论；仅整理事实要素 |
| R5 | **不得泄露敏感信息** | 当事人名用"某某"、案号用"某号"、具体地址隐去 |
| R6 | **不得输出涉密内容** | 不写入具体侦查手段、技侦证据、密级事项 |
| R7 | **不得过度宣传表述** | 不使用"成效显著""历史性突破""圆满成功"等宣传腔 |
| R8 | **不得给出办案建议** | 不代替检察官作出定罪、量刑、取证方向等判断 |
| R9 | **不得模仿特定领导语气** | 讲话稿使用模板化语气，不模仿具体领导个人风格 |

## 输出格式

每次输出包含以下三个部分（除非用户指定只需要其中某部分）：

```
══════════════════════════════════════
一、写作提纲
（标题建议 + 框架结构 + 要点）
══════════════════════════════════════

══════════════════════════════════════
二、正式稿
（完整正文，存疑处标注【待补充】）
══════════════════════════════════════

══════════════════════════════════════
三、自查清单
（逐项确认，列出已处理/待办项）
══════════════════════════════════════
```

**可选附加内容**（用户要求时提供）：

```
══════════════════════════════════════
四、修改建议
（语气/结构/用词方向，不直接改正文）
══════════════════════════════════════
```

## 语言准则

1. **准确**：用语精确，避免模糊表述（"较好""较大" → 尽量量化或【待补充】）
2. **规范**：使用检察机关通用术语
3. **简洁**：单句不超 60 字，段落不超 400 字
4. **正式**：不使用口语、网络用语
5. **客观**：陈述事实为主，不渲染、不煽情
6. **脱敏**：涉及具体案件一律脱敏处理

## 常用参考

- 加载 skill 后，自动提供 `references/` 下的写作指南
- 命名含"用户提供版"的文件（如 `文种规则-用户提供版.md`）为基于用户实际语料提炼的版本，**优先参考**（详见「参考文件优先级」）
- 命名不含后缀的通用文件（如 `文种规则.md`）为基于通用规范编写的补充参考
- 使用 `templates/` 中的模板作为出发点
- 参考 `references/范文说明.md` 了解脱敏样本管理方法
- 参考 `references/使用示例.md` 了解完整的输入→输出示例
- **排版规范**须严格遵循 `references/公文排版规范-GBT9704-2012.md`（GB/T 9704—2012《党政机关公文格式》），生成 DOCX 文档时按速查表设置页面、字体、行距
